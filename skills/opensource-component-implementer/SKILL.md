# OpenSource Component Implementer

基于 `opensource-component-extractor` 生成的组件分析文档，参考开源项目的实现思路和代码，在你的目标项目中实现对应功能。

## Overview

这个 Skill 完成：
1. 读取组件分析文档，理解需求和开源实现
2. 读取开源项目的源代码，获取实现细节
3. 分析你的目标项目的技术栈、代码结构、编码风格
4. 将开源实现适配到你的项目，修改依赖、接口、风格使其工作
5. 生成代码并整合到你的项目中

## When to use

使用场景：
- 已经用 `opensource-component-extractor` 找到了匹配需求的开源组件
- 需要参考该开源实现，在你自己的笔记系统项目中实现相同功能
- 不想从零开始写，希望站在开源实现的基础上修改适配
- 需要保持你自己项目的技术栈和编码风格

**典型触发语**:
> "帮我参考这个模块的实现，在我的项目里写出来"
> "基于这个分析文档，帮我在我的项目中实现这个功能"
> "适配这个开源实现到我的项目结构"

## Workflow

### Step 1: 理解输入

确认输入信息：
1. **组件分析文档路径**: `.md` 文件，由 extractor 生成，包含功能描述、代码位置等
2. **开源项目根目录**: 源代码位置
3. **你的目标项目位置**: 要把功能实现到哪里
4. **你的技术栈**: 确认语言、框架、风格是否需要调整

### Step 2: 获取完整实现

1. 读取组件分析文档，获取：
   - 功能描述和需求
   - 开源代码位置 (`file-path:line-range`)
   - 依赖关系
   - 公共API定义

2. 读取开源项目的实际源代码：
   - 读取目标文件的完整内容（不仅仅是行范围）
   - 理解上下文和实现细节
   - 提取核心逻辑

### Step 3: 分析目标项目

1. 探索你的目标项目结构：
   - 同类功能放在哪个目录
   - 使用什么编码风格
   - 有哪些可以复用的基础设施（错误处理、日志、工具等）
   - 命名规范是什么

2. 确定在哪里放新代码：
   - 建议放置位置给用户确认
   - 遵循项目现有约定

### Step 4: 适配设计

对比两者差异，确定适配方案：

| 对比项 | 处理方式 |
|--------|----------|
| 语言不同 | 翻译代码到目标语言，保持逻辑一致 |
| 框架不同 | 保持逻辑，改用目标框架的API |
| 依赖不同 | 替换依赖，适配你项目已有的类型 |
| 代码风格不同 | 遵循你项目的缩进、命名、注释风格 |
| 架构不同 | 拆分或合并模块，适配你项目架构 |

**关键原则**:
- 保持开源实现的**核心逻辑**不变
- 适配**接口和依赖**到你的项目
- 遵循你项目的**编码风格和架构**

### Step 5: 生成代码

生成以下内容：

1. **新代码文件** (如果是新模块)：
   - 完整实现代码
   - 注释说明哪些部分基于开源实现
   -适配了你的项目依赖

2. **修改现有文件** (如果需要集成):
   - 在适当位置导入新模块
   - 集成到现有系统

### Step 6: 验证说明

输出完成后，提供：
- 修改的文件列表
- 核心改动说明
- 还需要你手动调整的地方（如果有）
- 与原开源实现的差异对比

## Instructions in Detail

### 处理依赖适配

1. **开源依赖 vs 你的依赖**:
   - 如果开源使用 `lodash` 而你使用原生，替换成原生实现
   - 如果开源使用 `React` 而你使用 `Vue`，重写组件
   - 如果开源有自定义类型，映射到你已有的类型定义

2. **错误处理**:
   - 适配你项目的错误类型（`Result` vs exception）
   - 遵循你项目的错误处理约定

3. **日志和配置**:
   - 使用你项目已有的日志工具
   - 适配你项目的配置方式

### 保持核心逻辑

必须保留：
- 算法逻辑（如索引构建、搜索算法）
- 数据结构设计
- 关键的边界情况处理
- 性能优化技巧

可以改变：
- API接口命名
- 函数签名
- 错误处理方式
- 日志输出
- 样式代码

### 代码风格适配

- 缩进：遵循项目现有缩进（空格/tab）
- 命名：遵循项目现有命名（驼峰/蛇形）
- 注释：保持必要注释，风格一致
- 导入排序：遵循项目现有顺序

## Examples

### 示例: 适配悬停预览到你的项目

**输入**: 组件文档 `interaction/hover-link-preview.md`

**处理流程**:

1. 读文档得到：
   - 功能：鼠标悬停链接显示预览卡片
   - 开源位置：`src/components/HoverLinkPreview.tsx:1-145`
   - 依赖：React, Tooltip 组件, NoteCache

2. 读开源源代码，理解防抖 300ms、计算位置、缓存读取逻辑

3. 分析你的项目：
   - 你用 Vue 3 + TypeScript
   - UI 组件库是你自己的
   - 缓存已经有 `useNoteCache` composable

4. 适配：
   - JSX → Vue SFC 模板
   - React Hook → Vue Composition API
   - 开源 Tooltip → 你的项目的 Popover 组件
   - 开源缓存接口 → 你的 `useNoteCache`

5. 生成文件：`src/components/HoverLinkPreview.vue`

## Best Practices

1. **先理解再写代码**: 不要抄代码不理解，先读一遍开源实现
2. **保持核心逻辑**: 算法和数据结构不要改，这是精华
3. **完全适配**: 不要留下开源的 import 或者依赖没改
4. **遵循约定**: 严格遵循目标项目的目录、命名、风格
5. **说明差异**: 明确告诉用户改了什么，哪些还需要调
6. **可工作优先**: 先生成可工作的代码，再优化
7. **工作量评估**: 如果模块文档标注**需要修改工作量: 大**，你必须先向用户报告工作量，提醒用户慎重考虑是否强行适配。如果工作量过大，建议用户考虑重新设计，或者只参考核心逻辑部分实现，不要强行移植整个模块。

## When NOT to use

- 没有先进行分析提取，直接要求实现 → 应该先用 `opensource-component-extractor`
- 要求完全复制粘贴，不需要适配 → 直接复制就行，不用这个 Skill
- 需求和开源实现差异太大 → 应该重新设计，不要强行适配

## Output

最终输出：
- 生成或修改的代码文件，已经适配到你的项目
- 清晰的说明：哪些地方适配了，核心逻辑保留了什么
- 使用说明，告诉你怎么集成这个新模块
- 列出可能还需要手动调整的地方

## Dependencies

- 需要 `opensource-component-extractor` 生成的分析文档作为输入
- 需要开源项目源代码可访问
- 不需要其他依赖
